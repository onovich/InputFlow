# ADR 0008: v0.1 Release Candidate Policy

> Status: Accepted for Phase 9
> Date: 2026-06-22

## Context

Phase 8 established local validation, GitHub Actions workflow definitions,
browser smoke gates, release dry-run workflows, CI troubleshooting, and a
manual Gamepad release checklist. The remaining release-readiness gap is
evidence, packaging metadata, and public-facing review material.

InputFlow is not ready for a real publish workflow in Phase 9. The project still
needs reviewer-facing documentation, package metadata audit, tarball content
audit, remote GitHub Actions evidence, and a clear release candidate conclusion
before anyone creates a tag, GitHub Release, or npm package publish.

The Phase 9 question is therefore not "is v0.1 published?" It is "is the v0.1
release candidate ready for human release review?"

## Decision

Phase 9 will produce a v0.1 release candidate review package, not a real
release.

Phase 9 may:

- Observe required remote GitHub Actions runs for validation and Chromium smoke.
- Trigger or observe manual dry-run workflows when permissions allow.
- Improve README, changelog, release notes, package metadata, and package
  metadata guards.
- Run local `pnpm validate`, `pnpm browser:test`, `pnpm browser:test:all`,
  `pnpm release:dry-run`, and `pnpm package:dry-run`.
- Record package dry-run tarball contents and non-blocking npm warnings.
- Produce a final report with an explicit RC status.

Phase 9 must not:

- Run `npm publish`.
- Create a GitHub Release.
- Create or push a git tag.
- Require private npm or GitHub secrets.
- Add an automatic publish pipeline.
- Promote the optional Firefox / WebKit matrix to a required gate without a
  future ADR.
- Add Sinan, React diagnostics, rebind UI, mobile joystick, physical Gamepad
  automation, or other deferred product scope.

## RC Status Values

Phase 9 reports exactly one of these statuses:

- `RC_READY`: local validation, required remote validation, required remote
  Chromium smoke, remote release dry-run, package dry-run audit, public docs,
  and metadata guards all pass. The repository is ready for human v0.1 release
  review, but no release has been created.
- `RC_READY_LOCAL_ONLY`: local validation and local release confidence pass, but
  one or more remote manual or optional checks cannot be observed or triggered.
  The report must name the missing remote evidence. This status is not a remote
  release PASS.
- `BLOCKED`: a required remote gate cannot be observed, a required remote gate
  fails, remote release dry-run cannot be triggered when required for full RC,
  or another release-readiness blocker remains unresolved.

The final report must not claim `RC_READY` without run id or URL evidence for
required remote gates and the remote release dry-run.

## Evidence Requirements

Remote run evidence should include:

- Workflow file or workflow name.
- Run id.
- Branch.
- Commit SHA.
- Status and conclusion.
- URL.
- Observation time.

Package review evidence should include:

- Public package metadata audit results for `@inputflow/core`,
  `@inputflow/schema`, `@inputflow/testing`, and `@inputflow/browser`.
- `pnpm package:dry-run` result.
- Tarball contents summary.
- Any npm environment warning classified as blocking or non-blocking.

## Architecture Boundaries

Release candidate logic belongs in docs, workflow configuration, scripts,
package manifests, and reports. It must not enter the runtime hot path.

`@inputflow/core` must remain free of DOM, React, Three, Sinan, Zod hot-path,
Playwright, GitHub Actions, npm publish tooling, and release automation
dependencies.

Required validate, required Chromium smoke, release dry-run, and optional
browser matrix must remain separate gates with separate conclusions.

## Consequences

- Reviewers can evaluate v0.1 readiness without ambiguity about what was
  actually released.
- Remote CI evidence becomes a first-class release candidate artifact instead
  of an implied trust in local validation.
- Publish automation stays deferred until a later phase with explicit token,
  versioning, tagging, and release-notes policy.
- If remote evidence is unavailable, Phase 9 can still preserve useful local
  work while reporting `RC_READY_LOCAL_ONLY` or `BLOCKED` honestly.
